# Loop Engineering 个人方法论

## 一句话定义

Loop Engineering 是围绕一个可证据化目标，设计让 Agent 重复执行、评审、修正、记录和停止的工作系统，并把高频经验沉淀回 [[domains/AI Agent工程/03-Skill设计/00-Skill世界观|Skill]]、工具链和 LLM Wiki。

它不是“让 Agent 无限继续”，而是把人原本一轮轮说“继续、检查、再改、再跑一次”的过程，变成一个有目标契约、证据面、执行节点、评审节点、预算和停止条件的循环。

## 这个概念解决什么问题

传统 prompt engineering 主要回答“这一轮怎么问”。Loop engineering 回答的是“怎样让 Agent 围绕一个目标自己推进多轮，并且不会跑偏、失控或只是在自我感觉良好”。

这种模式已经在多个案例中被使用。下面这些案例只用于帮助 Agent 建立类比，不限定 Loop Engineering 的使用范围：

- 小工具开发：Codex 先实现，再从独立 agent 或 reviewer 视角审核，再回流修改。
- DMP 竞品分析：先挖 DMP AI 内部工具和数据，再生成 Excel/HTML 报告，再以运营视角评审，缺什么数据就回到采集阶段。
- 聚宽量化策略：先回测基线，再结合量化知识库提出修改，再回测验证，再判断是否真的改善 Sharpe、回撤和稳健性。

这些不是三个孤立工作流，也不是本方法论的边界，而是同一套 loop 思维在不同业务基座上的投影。未来任何需要“多轮执行、证据检查、评审回流、受控停止”的任务，都可以先按本方法论设计 loop，再决定具体领域知识和工具。

## Loop 的主线

```text
目标契约 -> 工作基座 -> 执行 -> 证据采集 -> 独立评审 -> 决策 -> 下一轮或停止 -> 总结入库
```

主线解释：

| 节点 | 作用 | 典型产物 |
| --- | --- | --- |
| 目标契约 | 说明最终要让什么变成真 | Goal、项目计划、成功标准 |
| 工作基座 | 说明 Agent 可以依赖什么 | repo、CLI、MCP、浏览器、账号、知识库、数据源 |
| 执行 | 生成代码、数据、报告、策略或视觉资产 | diff、JSON、Excel、HTML、策略代码 |
| 证据采集 | 证明这一轮发生了什么 | 测试、回测、截图、日志、数据表、指标 |
| 独立评审 | 用另一个视角判断是否达标 | review report、运营审阅、量化审计 |
| 决策 | 决定保留、回滚、补数据、继续或停止 | 下一轮计划、停止原因 |
| 总结入库 | 把经验写回知识系统 | wiki、skill patch、项目 README、报告 |

## Loop 的八个设计件

| 设计件 | 必须回答的问题 |
| --- | --- |
| 目标 | 最终要改善什么指标、资产、工具、报告或决策质量？ |
| 证据面 | 用什么证明目标达成：测试、benchmark、回测、截图、数据、人工 checklist 还是业务报告？ |
| 执行者 | 谁负责产出：主 Agent、subagent、CLI、脚本、浏览器自动化、平台 AI？ |
| 评审者 | 谁负责判断：另一个 Agent、同 Agent 的 reviewer 角色、脚本、用户、业务规则？ |
| 工具和权限 | Agent 可以用哪些目录、账号、API、MCP、CLI、网页、知识库？哪些动作必须确认？ |
| 迭代策略 | 失败后下一轮怎么选动作：补数据、修代码、换假设、改报告、回滚还是停止？ |
| 预算和停止 | 最多几轮、最长多久、多少 token、什么风险或证据缺失必须停？ |
| 留痕和回写 | 每轮记录在哪里，哪些经验进入 wiki、skill、模板或工具？ |

## Loop 类型

| 类型 | 触发方式 | 适合场景 | 风险 |
| --- | --- | --- | --- |
| Goal loop | 用户给定目标后持续推进 | 代码优化、策略调参、研究报告、多轮修复 | 目标太虚会漂移 |
| Heartbeat loop | 定期唤醒检查 | repo 维护、PR 巡检、监控任务 | 低价值检查会烧成本 |
| Cron loop | 固定时间执行 | 日报、周报、数据同步、竞品监控 | 需要明确数据源和失败告警 |
| Hook loop | 事件触发 | 提交后 review、文件变化后 lint、报告生成后截图检查 | hook 过多会干扰正常工作 |
| Evaluator-optimizer loop | 生成者和评审者交替 | 文案、代码、报告、翻译、UI、策略优化 | 同一模型自评容易宽松 |
| Orchestrator-workers loop | 中央 agent 拆分任务给多个 worker | 多文件代码修改、多源研究、复杂报告 | 协调成本和上下文成本高 |
| Telemetry/eval loop | 基于真实运行信号迭代 | agent 工具调试、生产系统优化、代码可靠性 | 没有 traces/evals 时会变成猜测 |

## 三个核心原则

### 1. 先定义证据，再开始循环

没有证据面，loop 会变成“更努力地讲故事”。以下只是案例化证据面，未来任务应根据自己的目标重新定义证据：

- 代码工具：单测、lint、真实 CLI 调用、跨目录调用、错误码、安装验证。
- DMP 运营：原始 JSON、Excel 指标、数据覆盖、报告可读性、运营动作建议。
- 聚宽量化：基线回测、同口径回测、年化收益、Sharpe、最大回撤、换手、分年度表现、样本外检查。
- 视觉生产：生成图、编辑 PSD、OCR 检查、尺寸、文字层、品牌规范、截图对比。

### 2. 执行者和评审者分离

写代码的 Agent 不应该独自宣布“代码很好”；写报告的 Agent 不应该只从文字通顺度判断报告；优化策略的 Agent 不应该只看 Sharpe 升高。

最低要求是角色分离：

```text
Builder: 负责产出
Reviewer: 负责按证据和 checklist 审核
Decider: 负责决定继续、回滚、补充数据或停止
```

更强的做法是工具分离：用测试、回测、截图、静态检查、Excel 校验、真实浏览器、平台导出作为外部证据，让评审不是纯文本意见。

### 3. Loop 必须能停止

Loop 的价值来自持续推进，但质量来自可停止。

停止条件至少包括：

- 成功：证据面显示目标达成。
- 轮数：达到用户指定的 5 轮、10 轮或其他上限。
- 预算：token、时间、API 调用、平台成本达到上限。
- 阻塞：缺权限、缺账号、工具失败、数据不可得。
- 风险：需要删除、发布、交易、发消息、授权、真实下单等高影响动作。
- 无收益：连续多轮没有指标改善，或改进只来自可疑过拟合。

## 案例库

本节是案例库，不是适用范围清单。Agent 遇到新领域时，不应问“是否属于下面这些场景”，而应问“这个任务的目标、证据面、执行节点、评审节点和停止条件分别是什么”。如果新任务无法归类到下列案例，也仍然可以使用 Loop Engineering。

### 小工具和 CLI 开发 Loop

主目标：让一个本机 CLI 工具达到可被 Codex、Hermes、SealSeek/OpenClaw 等 Agent 稳定调用的状态。

推荐流程：

1. 定义接口：命令、参数、输入输出、错误码、安装路径、跨 Agent 调用方式。
2. 构建：修改代码、脚本、文档和测试。
3. 本机验证：运行真实命令、单测、lint、权限检查。
4. 独立评审：从“另一个 Agent 要调用这个工具”的视角检查可用性。
5. 修复：只处理阻塞项、高风险项和高价值改进。
6. 总结：输出轮次报告、剩余风险、推荐写回的 skill 规则。

评审 checklist：

- 命令是否可发现、可复制、可在系统目录运行。
- 输出是否结构化，是否适合 Agent 解析。
- 错误信息是否足以让 Agent 自我修复。
- 是否有 destructive 操作保护。
- 是否有最小可运行示例。

### DMP 竞品分析 Loop

主目标：把 DMP AI 的内部工具能力转化为可复用的竞品数据采集和运营报告闭环。

推荐流程：

1. 工具发现：识别 DMP AI 内部工具、数据资产、VIEW、DEEPLINK、指标口径。
2. 数据采集：围绕竞品销售、搜索词、推广、人群、人群资产、店铺层数据生成 JSON/Excel。
3. 报告生成：生成 HTML 报告和 Excel 附件。
4. 运营评审：从“老板/运营负责人能不能据此行动”的视角检查。
5. 回流：报告缺洞时回到数据采集，不只是润色页面。
6. 交付：给出质量评分、数据覆盖、行动建议和下一轮挖掘问题。

评审 checklist：

- 是否回答了“竞品为什么卖得好/不好”。
- 是否有销售、搜索、推广、人群、店铺层之间的解释链。
- 是否区分事实、推断和建议。
- 是否保留原始数据和指标口径。
- 是否能直接导出给运营使用。

### 聚宽量化策略优化 Loop

主目标：在受控实验中提高策略质量，而不是只追逐一次回测 Sharpe。

推荐流程：

1. 基线回测：保存收益、风险、交易、持仓、回撤、Sharpe、分年度表现。
2. 诊断：读取量化知识库，判断收益假设、信号、仓位、执行成本、市场状态。
3. 单假设修改：每轮只改一个主要假设或一组强相关参数。
4. 同口径回测：记录指标变化和失败模式。
5. 反过拟合评审：检查是否牺牲回撤、换手、稳健性、可解释性或样本外。
6. 总结：列出有效修改、无效修改、保留代码、回滚代码、下一步实验。

评审 checklist：

- 是否保留原始基线。
- 是否同口径比较。
- 是否区分样本内、样本外、分年度表现。
- 是否记录交易成本和换手。
- 是否解释 Sharpe 改善来自收益、波动下降还是偶然年份。

## 成本和上下文控制

Loop 越长，越需要把状态从聊天上下文移到外部文件：

- `PLAN.md`：当前目标、边界、下一步。
- `ITERATIONS.md`：每轮动作、证据、评审、决策。
- `EVIDENCE/`：测试结果、截图、Excel、回测报告、日志。
- `FINAL_REPORT.md`：最终结论、质量评分、剩余风险。

控制方法：

- 每轮结束写状态文件，不把所有历史塞进上下文。
- 只给执行者当前轮需要的上下文。
- 让 reviewer 读取产物和证据，而不是完整聊天。
- 设置最大轮数和提前停止条件。
- 对高成本 subagent 使用“必要时调用”，不要默认每轮全量并行。

## 与 Skill 世界观的关系

Loop 是 Skill 的运行实验场。Skill 定义主线、节点、分支、工具、QA 和演化规则；Loop 让这些规则在真实项目中重复执行并暴露缺口。

稳定经验的回写方式：

- 高频成功路径：合并进 skill 主线。
- 特定输入变体：写入分支规则。
- 工具报错和平台变化：写入工具节点错误处理。
- 偶发但可复用经验：写入补丁区。
- 质量不稳定：补充 QA 和回主线检查。
- 新的证据标准：写入测试与回归规则。

## 最小可用 Loop 提示模板

```text
请先不要直接执行。先为这个任务设计一个 loop 项目。

目标：<我要改善或完成什么>
领域或任务：<代码工具 / 电商运营 / DMP / 聚宽量化 / 视觉制作 / 研究 / 文档 / 数据分析 / 其他>
输入：<已有文件、账号、网页、代码、数据、知识库>
最多轮数：<N>
成功证据：<测试 / 回测 / 报告 / 截图 / 数据 / 人工标准>
限制：<不能做什么，哪些动作要确认>

请输出：
1. Loop 名称
2. 目标契约
3. 工作基座
4. 每轮执行节点
5. 每轮评审节点
6. 回流规则
7. 停止条件
8. 每轮记录格式
9. 最终报告格式
10. 哪些经验要写回 wiki 或 skill
```
