# Skill 世界观

## 一句话定义

Skill 是大模型执行某类复杂任务的说明书。它定义目标、主线、执行方法、工具使用方式、异常处理原则和演化机制。它不是一次性 prompt，也不等同于固定工作流，而是一个能指导大模型围绕主线完成任务、处理意外并持续进化的小型执行知识体。

## 1. Skill 是大模型执行方式的说明书

Skill 首先是一份给大模型读取和执行的说明书。

它通常定义：

- 基本目标：这个 skill 要稳定完成什么任务。
- 执行方法：任务应该按什么顺序、什么判断方式推进。
- 工具使用：可能调用 Python/Node 脚本、CLI、MCP、浏览器、API 或其他本地工具。
- 边界条件：哪些事情可以自主判断，哪些必须询问用户，哪些必须停止。
- 输出要求：结果应该以什么结构、文件、报告或操作形式交付。

因此，skill 的核心不是“写一段提示词”，而是把一类任务的执行方式外化成可复用的操作说明。

## 2. Skill 也是一个定向的小型知识库

Skill 不只是流程说明，也承载定向知识。

这些知识可能来自 LLM Wiki、项目经验、工具调试记录、业务规则、历史踩坑、固定模板和脚本说明。设计完成后，skill 就成为一个相对独立的个体，负责执行某类固定任务。

它和 wiki 的关系是：

- Wiki 是更大的长期知识底座。
- Skill 可以从 wiki 中抽取某类任务所需的知识。
- Skill 一旦成型，就不应该每次运行都重新理解整个 wiki。
- Skill 内部应保留足够知识，让大模型能独立完成任务。
- 当 wiki 或实践经验更新后，skill 也应被重新补强。

所以，skill 是从知识库中长出来的执行型知识单元。

## 3. Skill 必须具备不断进化的能力

Skill 是说明书，而需求、工具、输入形态和异常情况都会变化。因此 skill 不能被设计成一次写死的文档。

它应该能随实践演化：

- 新需求出现时，扩展新节点或新分支。
- 新工具稳定后，补充工具调用规则。
- 新错误被解决后，把处理方法写成补丁。
- 某类补丁频繁出现后，在用户许可下合并进主流程。
- 旧规则失效时，更新约束和回退路径。

这意味着 skill 不是静态文档，而是可维护、可增量演化的执行知识体。

## 4. Skill 设计之初就要结构化

复杂 skill 往往对应的是一个工作流，而不是一个简单任务。简单任务通常不需要做成 skill。

既然 skill 承载的是复杂工作流，它就应该在设计之初被结构化：

- 有总纲：说明目标、主线、边界和设计原则。
- 有节点：把复杂任务拆成可识别的阶段。
- 有分支：处理不同输入、不同场景、不同用户目标。
- 有工具层：说明哪些节点用脚本、CLI、MCP 或 API。
- 有补丁区：记录运行中遇到的特殊情况和处理经验。
- 有 QA：检查结果是否回到主线，是否满足输出标准。
- 有演化规则：什么时候补丁继续保留，什么时候合并进主流程。

这种结构化的目的，不是让 skill 变得死板，而是让它有清晰骨架，方便未来插入节点、创建分支、增加补丁和持续升级。

## 5. Skill 像工作流，但不等同于工作流

Skill 和工作流有相似处：它们都处理多步骤任务，也都需要明确节点、输入、输出和顺序。

但两者的本质不同。

工作流更强调确定性。它通常假设输入符合预期，流程按固定路径运行。成功就是跑完，失败往往来自未覆盖的特殊场景，例如输入格式不同、数据不在设计范围内、某个工具返回异常。

Skill 更强调指导性。它要求大模型按照设计方向运行，但允许大模型在异常情况中自主判断、修复和调整。它不是把所有情况预先写死，而是给大模型一条主线和一组处理原则。

这种差异带来两个关键能力：

1. 单次运行中的自修复能力：遇到意外输入或工具异常时，大模型不至于立刻失败，而是尝试诊断、绕路、补救或询问。
2. 长期运行后的补丁能力：当大模型成功处理了某类意外情况，这个经验可以被写回 skill，成为以后处理类似问题的补丁。

因此，skill 的理想形态不是固定流水线，而是“主线清晰、局部弹性、经验可沉淀”的执行系统。

## 6. Skill 的补丁机制

补丁是 skill 进化的关键单位。

当某次运行中出现了设计之初没覆盖的情况，如果大模型成功处理了它，后续更新 skill 时不应立刻改写主流程，而应先把它作为补丁写入固定位置。

补丁应记录：

- 触发条件：什么输入、什么工具异常、什么业务场景会触发。
- 判断方法：如何识别这类情况。
- 处理动作：大模型应怎么补救或转向。
- 回主线位置：处理完后回到主流程的哪个节点。
- 是否可合并：如果高频发生，未来是否应进入主流程。

补丁的价值在于，它不会污染主线，又能让 skill 快速吸收真实运行经验。

如果某个补丁反复出现，并且已经成为常规场景，就可以在用户许可下把它合并进主流程。这样 skill 的主流程会逐步成熟，但不会因为少数异常而过早膨胀。

## 7. Skill 的核心风险：主线漂移

Skill 的优点是灵活，缺点也是灵活。

因为 skill 是给大模型看的说明书，而不是强制执行的程序，大模型可能没有按照设计本意运行。尤其当意外情况出现后，大模型容易临时发挥，进而偏离任务主线。

所以设计 skill 时必须注意两点：

1. 主线必须非常清晰。
   - 每个阶段都要知道自己服务于哪个总目标。
   - 每个分支都要说明何时进入、何时退出。
   - 每次异常处理后都要明确回到哪个主流程节点。

2. 异常处理必须能回主线。
   - 大模型可以灵活处理问题。
   - 处理过程可以产生补丁。
   - 但补丁不能替代主线。
   - 处理完必须回到原任务目标、原输出要求和原交付标准。

换句话说，skill 的弹性必须被主线约束。没有主线，skill 会漂移；没有弹性，skill 又会退化成脆弱工作流。

## 8. 工作流可以成为 Skill 的一个环节

Skill 和工作流不是互斥关系。

如果 skill 中存在非常重要、固定、可验证、不能靠模型自由发挥的流程，就应该用脚本或代码实现，而不是只写在说明书里。

适合代码化的部分包括：

- 文件解析和格式转换。
- 表格清洗和数据计算。
- API 调用和鉴权处理。
- 批量下载、上传、导出。
- 稳定的质量检查。
- 可确定的渲染、截图、diff、lint、校验。
- 高风险或不允许模型猜测的步骤。

在 skill 世界观里，工作流可以是 skill 的一个节点。大模型负责理解任务、调度流程、处理异常和回到主线；代码化工作流负责完成高确定性的局部任务。

因此，一个成熟 skill 往往是：

```text
说明书主线 + 定向知识库 + 工具/脚本节点 + 异常补丁 + QA 回路
```

## 9. 我的判断：这个理解是成立的

这个理解是成立的，而且比“skill 等于 prompt”或“skill 等于工作流”更接近真实可用的 Agent 工程。

原因有三点。

第一，它承认了大模型的优势。大模型擅长在不完美输入、不完整信息和临时异常中做判断、修复和转向。Skill 如果完全写成固定流程，就浪费了大模型的弹性。

第二，它承认了大模型的风险。大模型也容易漂移、误解、过度发挥或把补救动作当成新目标。因此 skill 必须有清晰主线、回主线机制和 QA。

第三，它把 skill 设计变成可演化工程。真实任务不会一次设计完美，最合理的做法是先建立主线，再用补丁吸收真实运行经验，最后把高频补丁合并进主流程。

这套世界观可以作为 后续设计复杂 skill 的总原则。

## 10. Skill 测试：防止上下文污染和假修改

Skill 的测试不能只看“当前对话里跑通了没有”。因为 skill 本质上是一段注入给大模型的说明书，它会和当前对话历史、用户补充、Agent 临时理解、未落盘修改一起发生作用。

因此，在设计和测试 skill 时，至少要防止两类假阳性：

1. 上下文污染：在设计 skill 的原对话中，Agent 已经知道很多背景、意图和临时修正，所以即使 skill 本身不完整，也可能跑出正确结果。
2. 假修改：用户要求优化 skill 后，Agent 可能只是依据当前对话手动补救，实际没有把规则写回 `SKILL.md` 或相关文件。当前对话看似成功，新对话重新运行就失败。

### 正确测试方法

一个更稳的 skill 测试流程应拆成四步。

第一步：设计会话内冒烟测试。

在创建或修改 skill 的同一对话中先跑一次测试，目标是快速发现明显缺口，例如触发条件不清、主线不完整、工具调用失败、输出格式不对。这个阶段的成功只能说明“当前上下文下可行”，不能说明 skill 已经合格。

第二步：落盘验证。

每次说“优化 skill”之后，都必须确认修改真的写入了文件。

最低检查包括：

- 明确被修改的文件路径。
- 查看 `git diff` 或文件内容片段。
- 确认新增规则进入了正确位置，例如主流程、分支、补丁区、工具说明或 QA。
- 如果新增了脚本、模板、reference，也要确认这些文件存在并被 `SKILL.md` 引用。

这一关专门防止 Agent 在对话中临时发挥，而没有真正更新 skill。

第三步：干净会话回归测试。

在新对话中重新触发 skill，并尽量只提供真实用户以后会提供的信息，不带原设计过程中的解释和背景。这个测试用来验证 skill 是否能脱离设计会话独立工作。

干净会话测试时应注意：

- 不提示 Agent 刚才修过什么。
- 不补充只有设计会话中才知道的背景。
- 使用真实输入样本或边界样本。
- 观察 Agent 是否自动读到了 skill，并沿着主线执行。
- 检查输出是否满足 skill 自己定义的目标和 QA。

第四步：问题回写。

如果干净会话失败，不要只在新对话里手动修好结果，而要把失败原因回写到 skill。

回写时先判断失败属于哪一类：

- 主线缺失：合并进主流程。
- 输入变体：加入分支规则。
- 工具异常：加入工具节点的错误处理。
- 偶发特殊场景：加入补丁区。
- 输出不稳定：加强 QA 和输出模板。
- Agent 偏离本意：补充回主线检查。

修完后重新走“落盘验证 + 干净会话回归测试”。

### 测试通过的标准

Skill 不能以“某一次跑出了正确结果”为通过标准，而应以“离开设计对话后仍能稳定复现”为通过标准。

一个复杂 skill 至少要满足：

1. 原对话冒烟测试能跑通。
2. 文件确实落盘，diff 能看到规则变化。
3. 新对话在干净上下文中能触发并执行。
4. Agent 遇到异常后能补救，并回到主线。
5. 测试暴露的问题被写回 skill，而不是只靠当前对话补救。

这套测试方法的本质，是把 skill 从“对话中临时有效”推向“作为独立个体稳定有效”。

## 11. 设计复杂 Skill 的推荐骨架

以后创建复杂 skill 时，可以默认采用以下结构：

```markdown
# Skill 名称

## 使命与边界
## 主线流程
## 节点定义
## 分支规则
## 工具与脚本节点
## 异常处理原则
## 补丁区
## 测试与回归规则
## QA 与回主线检查
## 演化规则
```

其中最重要的是三件事：

1. 主线流程必须清楚。
2. 异常处理必须能回到主线。
3. 补丁必须有固定位置，不能随意污染主流程。
