# 商业模式画布九个构造块

## 问题定义

商业模式画布解决的是“大家到底在讨论同一个业务吗”。一个业务不是单独的产品、渠道、品牌口号、供应链或利润表，而是一组互相牵动的假设：服务谁，提供什么价值，如何触达和维系，靠什么收钱，用哪些资源、活动和伙伴交付，最终形成什么成本结构。

画布的价值在于把业务拆成九块，同时保持整体性。任何一块改变，都可能要求其他块同步改变。

## 核心模型

| 构造块 | 要回答的问题 | 诊断重点 |
| --- | --- | --- |
| 客户细分 | 我们为谁创造价值？谁是最重要客户？ | 不同客户是否真的有不同需求、渠道、关系和付费逻辑。 |
| 价值主张 | 我们帮客户解决什么问题，满足什么需求？ | 价值是否足以让客户从现有方案转换。 |
| 渠道通路 | 客户如何知道、评估、购买、获得和接受售后？ | 渠道组合是否兼顾触达、体验、成本和利润。 |
| 客户关系 | 客户期待什么关系？关系成本如何？ | 关系服务于获客、留存还是增购。 |
| 收入来源 | 客户为什么付费、如何付费、付多少？ | 收入机制是否与价值和客户习惯匹配。 |
| 关键资源 | 交付价值需要哪些核心资源？ | 资源是否真的不可缺，是否能自有或外部获取。 |
| 关键业务 | 交付价值必须做哪些关键活动？ | 活动是否支撑价值主张，而不是组织惯性。 |
| 重要伙伴 | 谁能帮助降本、降险、补资源或补活动？ | 伙伴关系是优化、联盟、合作竞争还是合资。 |
| 成本结构 | 模式运行会产生哪些核心成本？ | 成本驱动还是价值驱动，固定/可变成本与规模/范围经济。 |

## 操作流程

1. 先写客户细分和价值主张，不要先写产品功能。
2. 对每个客户细分分别写渠道、客户关系和收入来源，避免把多个客户混成一个平均客户。
3. 从价值主张倒推关键资源、关键业务和重要伙伴。
4. 用成本结构检查模式是否承担得起。
5. 回到收入来源，判断价值、价格、付费方式和客户预算是否闭环。
6. 标出风险最高的三块，进入验证或重构。

## 判断标准

一个画布有用，至少满足四个条件：

- 客户侧清楚：客户细分、价值主张、渠道、关系、收入之间能互相解释。
- 后台侧清楚：资源、活动、伙伴不是泛泛罗列，而是服务某个具体价值主张。
- 财务侧清楚：收入机制和成本结构之间存在可持续差额。
- 牵动关系清楚：改价格、改渠道、改客户群或改伙伴时，能看到其他块怎么变。

## 常见错误

- 把客户细分写成“所有消费者”。
- 把价值主张写成产品功能清单，没有说明客户为什么转换。
- 只写渠道名称，不写认知、评估、购买、交付、售后五阶段。
- 客户关系只写“私域/社群/客服”，没有说明它服务获客、留存还是复购。
- 收入只写“卖货”，忽略使用费、订阅、租赁、授权、经纪、广告等机制。
- 关键资源和关键业务写成公司已有部门，而不是模式必须具备的能力。
- 成本结构只看采购成本，忽略渠道、服务、库存、资金、组织和伙伴成本。

## 案例与证据锚点

iPod/iTunes 的价值不只是播放器，而是硬件、软件、内容授权、购买体验和生态联动形成的整体模式。Rolls-Royce 的航空发动机案例说明，价值主张可以从“卖产品”转成“帮客户完成任务”，收入从资产销售转成使用计费。Google 的搜索广告说明，同一个模式可能同时服务多个客户细分。

## Agent 使用提示

当用户给出一个业务想法时，先不要急着给增长动作。先输出一张九块画布，标注每块的确定性、证据和待验证假设。若用户只给产品或品牌信息，主动补问客户、渠道、收入和成本；若用户只给财务信息，主动补问价值主张和交付系统。
