# MVP发布与客户满意循环

## 问题定义

MVP 发布不是媒体发布会，也不是把半成品扔给所有人试用。更好的策略是把产品精准发布给早期采用者，验证产品是否兑现独特价值主张，并让客户进入激活和留存。

来源锚点：`raw/books/jingyi-chuangye-shizhan-3rd-2023/chapters/ch041-12.md`，`ch042-13.md`。

## 核心模型

### MVP 的目标

MVP 是以最小解决方案创造、交付并获得客户价值。它应验证：

- 客户是否能得到承诺中的结果；
- 客户是否愿意持续使用；
- 交付是否可重复；
- 客户工厂指标是否前进。

### 客户满意循环

获客只是第一场战斗。客户满意循环的核心顺序是：

1. 激活：客户第一次真正获得价值，迎来顿悟时刻；
2. 留存：客户再次使用并持续接近期望结果；
3. 变现：客户愿意持续付费或贡献价值；
4. 推荐：客户把新客户带进客户工厂。

激活和留存通常比变现更早暴露问题。即使你预先收费，如果客户没有获得价值，也会退款、流失或沉默。

## 操作流程

### 1. 固定发布日期

在解决方案设计冲刺之后，争取用约 2 个月准备 MVP，并向早期采用者宣布发布日期。外部承诺能限制团队不断加功能。

### 2. 控制范围

MVP 只服务使用前 90 天的关键任务。不要把远期愿景、全功能平台和边缘需求塞进去。持续交付比一次性完美更重要。

### 3. 分批发布

第一批客户应精心挑选：

- 动机强；
- 问题真实；
- 愿意反馈；
- 能接受高接触服务；
- 代表目标早期采用者。

朋友或友好客户可以用于降低启动阻力，但不能让反馈失真。

### 4. 建立客户工厂报表

把客户工厂的宏观事件映射到具体行为：

| 客户工厂步骤 | MVP 行为指标示例 |
| --- | --- |
| 获客 | 预约、注册、提交资料、支付订金 |
| 激活 | 完成第一次关键任务、看到核心结果 |
| 留存 | 再次使用、进入第二个周期、复购 |
| 变现 | 付费、升级、续费、扩大订单 |
| 推荐 | 引荐、分享案例、邀请同事/朋友 |

避免虚荣指标，例如累计注册人数、总浏览量、总下载数。可执行指标要能把重复行为和结果连接起来，最好按批次观察。

### 5. 设计顿悟时刻

客户第一次体验中，应尽量在短时间内让客户看到“原来这能帮我完成任务”。书中建议争取在 30 分钟内创造第一个顿悟时刻；这个数字是启发式标准，实际要按产品复杂度调整。

设计原则：

- 优先结果，不优先功能展示；
- 让客户实践，而不是学习一堆说明；
- 限制每个阶段可用功能；
- 把大任务拆成多个小顶点；
- 用进展路线图让客户知道自己走到哪里。

### 6. 对抗惯性力

客户购买后，转换动机很快会衰减。要通过行为设计建立新习惯：

- 正确设定期望；
- 帮客户创建提醒；
- 分享最佳实践；
- 在完成前一项任务后提示下一步；
- 定期接触；
- 针对行为发送鼓励；
- 借用客户已有习惯；
- 让反馈变得简单；
- 庆祝客户胜利。

### 7. 遵守发布后二八定律

发布后大部分时间应评估和改进现有功能与客户满意循环，而不是追逐新功能。客户提出功能要求时，要回到问题和任务判断，而不是机械照单全收。

## 判断标准

MVP 发布有效的证据：

- 早期客户按预期完成关键行为；
- 激活率和留存率可分批观察；
- 客户能说出产品帮他完成了什么任务；
- 团队知道哪里卡住，而不是只知道“数据不好”；
- 改进动作能改变下一批客户结果；
- 客户愿意续费、复购、扩大使用或推荐。

## 适用边界

实体产品、电商新品和服务同样可以使用 MVP 思路，但“持续交付”和“发布周期”要结合打样、生产、物流、平台审核和售后风险调整。对于强监管或安全风险产品，MVP 不能牺牲合规和安全。

## Agent 使用提示

Agent 设计 MVP 时，要输出：

- 早期采用者名单标准；
- MVP 前 90 天任务范围；
- 顿悟时刻；
- 客户工厂行为指标；
- 分批发布计划；
- 不做清单。
