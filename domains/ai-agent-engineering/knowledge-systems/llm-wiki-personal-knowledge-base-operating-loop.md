# LLM Wiki 个人知识库运行闭环

## 一句话定义

个人知识库之所以容易烂尾，不是因为使用者不自律，而是因为摘要、打标签、补引用、更新旧笔记这些高频整理动作长期由人承担，维护成本过高。LLM Wiki 的价值，在于把这些重复而枯燥的整理工作转交给 AI，让知识库从“人工记录系统”变成“AI 持续维护的知识复利系统”。

## 为什么传统知识库容易烂尾

传统笔记工具的失败点通常不在记录本身，而在后续维护：

1. 每读完一篇内容，都要自己提炼摘要。
2. 需要手动补标签和分类。
3. 要回头寻找旧笔记，补交叉引用。
4. 随着内容变多，标签体系和目录都会持续膨胀。
5. 这些动作没有明确终点，会形成持续累积的知识债务。

因此，真正导致烂尾的，不是信息摄入不足，而是知识整理工作长期由人手工承担。

## LLM Wiki 与 RAG 的本质差异

传统 RAG 的工作方式，是在提问时从原始资料中临时检索若干片段，再拼装出一个回答。这样可以快速回答问题，但知识不会沉淀，模型每次都在重新发现旧知识。

LLM Wiki 的工作方式则不同。它会把新资料先处理成持久知识页，再把这些知识页作为未来查询和更新的基础。这样一来，知识不是在每次提问时被临时取用，而是在每次 ingest 后被重新编译、重组和积累。

可以把两者理解为两种完全不同的知识机制：

1. RAG 更像每次去矿山临时挖矿。
2. LLM Wiki 更像把矿石熔炼成金属，并永久储存在自己的材料库中。
3. 新资料进入后，旧知识会被再次修订，而不是被丢在原地等待下次检索。

## 三层结构

### 1. 原始资料层

这一层保存文章、PDF、网页、视频字幕、剪藏内容等原始输入。核心原则是只读不改，保证所有结论都能回溯到原始来源。

### 2. AI 维护的 Wiki 层

这一层由 AI 持续更新，承载人物页、概念页、摘要页、对比页、工作流页等正式知识页面。所有整理、合并、补链、修订、去重、冲突处理主要发生在这一层。

### 3. 规则配置层

这一层负责约束 Agent 的行为，使不同会话、不同 Agent、不同时间段都遵守一致的结构与更新规则。视频中用 `CLAUDE.md` 举例，本质上它代表的是“系统说明书”。在 Peter 当前的 Wiki 中，这个职责更适合由 [[AGENTS]] 与 [[SCHEMA]] 共同承担。

## 日常运行闭环：Ingest、Query、Lint

### Ingest

Ingest 的目标不是简单保存资料，而是让一份新来源触发整个知识系统的更新。典型动作包括：

1. 保存原始资料到 `raw/`。
2. 提取其中有长期价值的知识。
3. 更新相关旧页面，而不是只新建孤立笔记。
4. 补充交叉引用、摘要页、概念页或实体页。

一篇新文章进入后，理论上可以触发多个已有页面的修订，这正是知识复利的开始。

### Query

Query 不是在原始资料中随机翻找，而是优先读取已经整理好的 wiki 页面。这样得到的回答更稳定，也更容易进一步沉淀为新的知识页。

高价值问答应反向写回知识库，因为提问本身就是知识系统继续长大的触发器。

### Lint

Lint 用来做知识库健康检查，主要负责发现：

1. 相互矛盾的页面。
2. 孤立的页面。
3. 过时的信息。
4. 缺少索引、缺少引用、缺少结构规范的页面。

Lint 的意义，不是修文字，而是持续降低知识系统内部的熵。

## 推荐工具栈

一个可落地的 LLM Wiki 个人系统，至少需要三类工具：

1. 本地 Markdown 编辑与浏览工具，例如 Obsidian。
2. 原始资料摄入工具，例如 Obsidian Web Clipper。
3. 能够读写本地文件并执行流程的 Agent 端，例如 Claude Code、Hermes、Codex CLI、OpenClaw 一类系统。

其中，Obsidian 负责浏览、编辑、图谱和剪藏；Agent 负责整理、链接、更新和维护；Git 负责同步、回滚和历史追踪。

## 对 Peter 当前 Wiki 的直接启发

这套方法对 Peter 当前个人 Wiki 的直接启发主要有五点：

1. 不要把知识库当作“资料堆”，而要把它当作“AI 持续维护的正式知识层”。
2. 抖音、课程、文章、网页都只决定 raw 的保存方式，不决定正式知识的最终归档位置。
3. 规则文件应作为系统级约束存在，保证多个 Agent 在同一库中协作时行为一致。
4. Query 和 Lint 也应被视为知识生产动作，而不只是辅助动作。
5. 知识库的核心价值不是检索速度，而是随着新资料进入，旧知识也会被持续修订和强化。
