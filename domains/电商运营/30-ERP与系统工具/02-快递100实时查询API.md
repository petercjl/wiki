# 快递100实时查询 API

## 用途

快递100实时查询接口用于按快递公司编码和运单号主动查询最新物流轨迹，返回当前物流状态、轨迹节点、节点时间、状态名称和可选的高级状态。它适合给 [[domains/电商运营/30-ERP与系统工具/01-旺店通开放平台API/index|旺店通]] 出库单补足“揽收、在途、派件、签收、疑难”等轨迹状态。

这个接口是主动查询接口，不适合对同一单号高频轮询。官方文档要求同一单查询频率至少间隔半小时，否则可能造成锁单。

## 接口契约

| 项目 | 值 |
| --- | --- |
| 请求地址 | `https://poll.kuaidi100.com/poll/query.do` |
| 方法 | `POST` |
| Content-Type | `application/x-www-form-urlencoded` |
| 认证参数 | `customer`、`sign`、`param` |
| 默认签名 | `MD5(param + key + customer)`，结果转 32 位大写 |
| 可选签名算法 | `MD5`、`SHA256`、`SM3`、`SM3-HMAC`，通过 `signType` 指定 |

`param` 是一个 JSON 字符串。签名时使用这个 JSON 字符串原文拼接 `key` 和 `customer`，不要额外加入加号。

## 请求参数

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| `customer` | 是 | 企业版授权码，需要在快递100申请企业版获取。 |
| `sign` | 是 | 身份签名。 |
| `signType` | 否 | 不传时默认 MD5。 |
| `param.com` | 是 | 快递公司编码，小写字母，例如 `shentong`、`yuantong`。需从快递100公司编码表映射。 |
| `param.num` | 是 | 运单号，长度 6-32 个字符。 |
| `param.phone` | 条件必填 | 收、寄件人的电话号码只能填一个；顺丰速运、顺丰快运、中通快递必填，其他快递选填。电商虚拟号码按官方规则可传 `-` 后四位。 |
| `param.from` | 否 | 出发地，标准省市区格式；有助于提高签收状态判断。 |
| `param.to` | 否 | 目的地，标准省市区格式；`resultv2=8` 时必填。 |
| `param.resultv2` | 否 | `1` 返回状态名称；`4` 返回高级状态、状态值、出发/目的/当前城市；`8` 在 `4` 基础上返回预计到达时间和预计轨迹。 |
| `param.show` | 否 | 返回格式，`0` JSON，`1` XML，`2` HTML，`3` text；默认 JSON。 |
| `param.order` | 否 | `desc` 倒序，最新轨迹在前；`asc` 正序。 |
| `param.lang` | 否 | `zh` 中文，`en` 英文。 |
| `param.needCourierInfo` | 否 | 为 true 时尝试提取快递员姓名和电话。当前物流监控不需要人员信息，默认不要开启。 |

## 推荐请求模板

```json
{
  "com": "shentong",
  "num": "770323466235644",
  "resultv2": "4",
  "show": "0",
  "order": "desc",
  "lang": "zh"
}
```

Node.js 签名骨架：

```js
import crypto from "node:crypto";

function kuaidi100Sign(param, key, customer) {
  return crypto
    .createHash("md5")
    .update(`${param}${key}${customer}`, "utf8")
    .digest("hex")
    .toUpperCase();
}
```

## 响应字段

| 字段 | 含义 | 监控用途 |
| --- | --- | --- |
| `state` | 当前基础物流状态。 | 判断是否揽收、在途、派件、签收、疑难等。 |
| `status` | 通讯状态，官方建议忽略。 | 不作为业务状态。 |
| `ischeck` | 是否签收标记，官方建议明细以 `state` 为准。 | 辅助字段，不作为唯一依据。 |
| `com` | 快递公司编码。 | 回写或核验公司映射。 |
| `nu` | 运单号。 | 和旺店通 `logistics_no` 对账。 |
| `data[]` | 轨迹节点数组，默认倒序。 | `data[0]` 是最新轨迹节点。 |
| `data[].context` | 轨迹文本。 | 展示异常原因或客服备注。 |
| `data[].time` / `ftime` | 轨迹时间。 | 计算发货后多久未揽收、在途多久未更新。 |
| `data[].status` | 节点状态名称，`resultv2=1/4/8` 返回。 | 用中文状态给报表分组。 |
| `data[].statusCode` | 节点高级状态值，`resultv2=4/8` 返回。 | 建议用于异常规则判断。 |
| `routeInfo` | 出发地、当前地、目的地行政区信息，`resultv2=4/8` 返回。 | 判断停留城市或当前位置。 |
| `arrivalTime`、`totalTime`、`remainTime`、`probability` | 预计到达相关字段，`resultv2=8` 返回。 | 时效预测，不是基础监控必需。 |
| `predictedRoute[]` | 预计轨迹节点，`resultv2=8` 返回。 | 高级 ETA 场景使用。 |

## 状态映射

| 基础状态值 | 状态 | 监控解释 |
| --- | --- | --- |
| `1` | 揽收 | 快递公司已进入揽收链路。高级状态包括 `101` 已下单、`102` 待揽收、`103` 已揽收。 |
| `0` | 在途 | 快件运输中。高级状态包括 `1001` 到达派件城市、`1002` 干线、`1003` 转递。 |
| `5` | 派件 | 正在派件。高级状态 `501` 表示投柜或驿站。 |
| `3` | 签收 | 已签收。高级状态包括本人签收、代签、投柜或站签收等。 |
| `2` | 疑难 | 异常或超时。重点关注 `201` 超时未签收、`202` 超时未更新、`204` 派件异常、`205` 柜或驿站超时未取、`208` 滞留。 |
| `6` | 退回 | 快件返回发货人途中。 |
| `4` | 退签 | 退签或撤销，含 `401` 已销单、`14` 拒签。 |
| `7` | 转投 | 转给其他快递公司邮寄。 |
| `8/10/11/12/13` | 清关相关 | 跨境或清关链路状态。 |

## 用于订单物流监控的规则

1. 先用旺店通 `stockout_order_query_trade.php` 或 `trade_query.php` 拉取已发货订单的 `order_no`、`consign_time`、`logistics_name`、`logistics_no`。
2. 将旺店通物流名称映射为快递100 `com` 编码。公司编码表需要从快递100开放平台下载维护。
3. 对每个运单调用实时查询，默认传 `resultv2=4`、`order=desc`、`lang=zh`。
4. 若 `state=1` 且最新高级状态是 `102`，可判定为“已发货但仍待揽收”。
5. 若旺店通已发货超过设定小时数，但快递100返回 `500` 无结果，或长期没有 `data[]`，可列为“发货后无轨迹”。
6. 若 `state=2` 或最新 `statusCode` 属于 `201/202/204/205/208`，列为“疑难/超时/滞留”。
7. 若 `state=0` 且最新节点距离当前时间超过内部 SLA，列为“在途长时间未更新”。
8. 若 `state=5` 且派件时间超过内部 SLA 仍未 `state=3`，列为“派件后未签收”。
9. 若 `state=3`，记录签收时间，可从最新签收节点或 `data[]` 中的签收节点提取。

## 错误码处理

| 信息代码 | 含义 | 处理 |
| --- | --- | --- |
| `200` | 查询成功。 | 解析 `state` 和 `data[]`。 |
| `400` | 找不到对应公司、数据不完整或账号未充值。 | 检查表单格式、公司编码、账号单量。 |
| `408` | 快递公司参数异常，验证码错误。 | 多见于手机号校验失败，检查 `phone`。 |
| `500` | 查询无结果。 | 确认快递公司、单号、是否已发货；可间隔后重查。 |
| `501` | 服务器错误或请求不规范。 | 检查参数格式；必要时重试或联系快递100。 |
| `502` | 服务器繁忙。 | 稍后重试。 |
| `503` | 验证签名失败。 | 检查 `param + key + customer` 顺序、算法和大写。 |
| `601` | key 已过期或无可用单量。 | 账号充值或更换有效 key。 |

## 隐私与边界

- 当前监控目标不需要收件人信息。除非快递公司强制，默认不要传 `phone`、不要开启 `needCourierInfo`。
- 顺丰速运、顺丰快运、中通快递按官方文档需要 `phone`，可以优先尝试寄件人电话或合规的最小可用信息。
- 不要把 `KUAIDI100_CUSTOMER`、`KUAIDI100_KEY` 写入 wiki、日志或导出表。
- 公司编码表未在本次 ingest 中归档，后续接入时需要单独下载并维护旺店通物流名称到快递100编码的映射表。
