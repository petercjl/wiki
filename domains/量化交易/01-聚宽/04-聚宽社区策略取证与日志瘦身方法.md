# 聚宽社区策略取证与日志瘦身方法

## 一句话结论

聚宽社区策略入库时，日志不应该承载全部结果数据。收益、交易、持仓应由聚宽导出的结构化文件负责；日志只负责解释关键决策路径：策略版本、候选池摘要、目标选择、调仓计划、订单结果、风险触发和调仓后状态。

## 为什么要瘦身日志

QMT四季发财v3 的第一轮长周期日志给出了一个明确教训：

- r2 日志解压后约 111MB、99828 行，被聚宽“最多只能导出100000条日志”限制截断，只覆盖 2021-02-24 之后。
- 主要膨胀来源是逐 ETF 的 `candidate_metric` 和 `candidate_rejected`。
- 截断后，即使收益、交易、持仓完整，也无法完整解释早期执行路径。

r3 改为 summary-only 后：

- 日志解压后约 18MB、19867 行。
- `JQ_AUDIT` 9503 条。
- 全周期未截断。
- HTML 反转义后 JSON 解析失败为 0。
- 仍能解释每日候选、目标、订单和组合状态。

## 四件套分工

| 导出/日志 | 负责回答 | 不应承担 |
| --- | --- | --- |
| 收益概述 CSV | 净值、总收益、年化、波动、回撤、日收益、年度/月度表现 | 逐候选指标解释 |
| 交易详情 CSV | 买卖次数、成交额、手续费、平仓盈亏、订单状态 | 策略为何选中某标的 |
| 持仓收益 CSV | 每日持仓、市值、现金、持仓收益 | 候选池过滤原因 |
| `JQ_AUDIT` 日志 | 版本、参数、候选摘要、目标选择、调仓计划、订单决策、风险触发 | 全量指标明细、完整结果报表 |

这个分工能让日志既可解析，又不压垮聚宽导出上限。

## 推荐 JQ_AUDIT 事件合同

默认保留：

| 事件 | 频率 | 最小内容 |
| --- | --- | --- |
| `strategy_metadata` | 初始化 1 次 | 策略名、版本、参数、ETF 池、过滤器开关、日志策略 |
| `daily_position_check` | 每日 1 次 | 总资产、现金、持仓、回撤、收益、基准快照 |
| `rebalance_universe_summary` | 每日调仓前 1 次 | ETF 池大小、通过数量、过滤计数、少量拒绝样本、前 3 名候选 |
| `target_selection` | 每日 1 次 | 正常/防御/空仓模式、目标 ETF、第一名候选、选择原因 |
| `rebalance_plan` | 每日 1 次 | 当前持仓、目标 ETF、目标金额 |
| `position_adjustment_needed` | 仅需要调仓时 | 当前市值、目标市值、偏离、容差 |
| `order_decision` | 下单或跳过时 | 停牌、涨跌停、T+1、min_money、目标数量、是否发送订单 |
| `order_result` | 订单返回时 | 买/卖、数量、金额、价格、是否成功、下单后现金和持仓 |
| `risk_event` | 仅触发时 | 固定止损、ATR 止损、触发价、成本价、持仓数量 |
| `after_rebalance` | 每日调仓后 1 次 | 总资产、持仓、现金、回撤、收益、目标 ETF |

默认删除：

- 逐 ETF `candidate_metric`。
- 逐 ETF `candidate_rejected`。
- 调仓前 `before_rebalance` 快照。
- 正常未触发 ATR 止损的每日 `risk_check`。
- 5% 容差内无需调仓的 `position_adjustment_skipped`。

## 解析注意事项

聚宽导出的日志可能把 JSON 双引号转义为 `&quot;`。本地解析时应先执行 HTML unescape，再 `json.loads`。

伪流程：

```python
import html
import json

if 'JQ_AUDIT|' in line:
    payload_text = line.split('JQ_AUDIT|', 1)[1]
    payload = json.loads(html.unescape(payload_text))
```

不要把“原始 JSON 解析失败”直接判定为日志坏了；先检查是否存在 HTML 转义。

## 入库判断清单

一个社区策略适合进入正式知识库，需要同时满足：

- 策略代码有原始归档或明确来源。
- 只做日志增强或明确记录了任何语义变更。
- 聚宽回测导出了收益概述、交易详情、持仓收益和日志。
- 日志没有被 100000 条上限截断。
- `strategy_metadata.strategy_version` 与本地脚本版本一致。
- 日志能解析出目标选择、订单结果和调仓后状态。
- 回测指标能从结构化导出复算或核对。
- 发现的策略 bug、订单失败、现金不足、停牌/涨跌停等执行问题都被记录为边界，而不是被收益结论掩盖。

## Agent 使用模板

当 Peter 给出一个新的聚宽社区策略时，Agent 应按此顺序工作：

1. 保存原始脚本和策略工作区。
2. 不改变策略语义，只增加 summary-only 审计日志。
3. 本地运行 `py_compile` 和 `check_joinquant_strategy.py`。
4. 让 Peter 在聚宽回测并导出四件套。
5. 先检查日志完整性、版本、JSON 解析和异常行。
6. 用收益概述计算绩效，用交易详情计算平仓盈亏和交易成本，用持仓收益核对日持仓。
7. 只有在数据足够时，才创建策略案例页。
8. 如果日志不够，先回到脚本改日志，不要勉强入库。
