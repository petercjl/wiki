# 交易执行、成本与市场冲击

执行层把目标仓位转成订单和成交。很多策略在信号层有效，但在执行层死亡：换手太高、流动性不足、滑点过大、成交失败、市场冲击被低估。

Backtrader 的 broker simulation 包含订单类型、佣金、滑点、成交量填充等；Qlib 也把 order execution 放在完整量化链路中。这说明执行不是细枝末节，而是策略收益能否落地的关键。

## 成本类型

| 成本 | 说明 | 回测处理 |
| --- | --- | --- |
| 佣金/手续费 | 券商或交易所收取 | 固定比例或每笔费用。 |
| 印花税/交易税 | 市场制度成本 | 按买卖方向设置。 |
| 买卖价差 | 以买价卖、卖价买的隐含成本 | 用 bid/ask 或滑点近似。 |
| 滑点 | 预期价格与实际成交价格差 | 按比例、固定 tick、成交量模型。 |
| 市场冲击 | 大订单推动价格 | 与订单规模/成交量相关。 |
| 借券/融资成本 | 做空或杠杆成本 | 按持仓天数计。 |
| 折溢价 | ETF/基金价格偏离净值 | 需要单独数据或保守滑点。 |

## 订单类型

| 订单 | 用途 | 风险 |
| --- | --- | --- |
| 市价单 | 确保快速成交 | 价格不确定，滑点大。 |
| 限价单 | 控制成交价格 | 可能不成交。 |
| 止损单 | 价格触发退出 | 跳空时成交差，日线回测不精确。 |
| 止盈单 | 锁定收益 | 可能过早退出趋势。 |
| 跟踪止损 | 跟随趋势保护利润 | 参数敏感。 |
| OCO/括号单 | 同时设置止盈止损 | 平台支持和撮合规则不同。 |

聚宽等平台是否支持某类订单，要回到 [[domains/量化交易/01-聚宽/02-聚宽API能力地图|聚宽 API 能力地图]] 核对。

## 换手率与容量

换手率决定成本敏感性。一个高 IC 信号如果每天换仓，可能被成本吃掉；一个低频信号虽然预测力弱，但可能净收益更好。

容量是策略能承载多少资金。容量受以下因素影响：

- 标的成交额和盘口深度。
- 单日下单占成交量比例。
- 持仓集中度。
- 调仓同步性。
- 是否需要做空或杠杆。
- 市场压力期流动性变化。

## 执行层检查

每个回测报告都应检查：

1. 成本前收益和成本后收益。
2. 年化换手和单日最大换手。
3. 手续费占收益比例。
4. 订单失败原因。
5. 下单时间是否与信号可见时间匹配。
6. 单标的成交额是否足以支持订单。
7. 是否存在涨跌停、停牌、不可成交。

## 对买卖器的理解

买卖器不是为了把曲线调漂亮，而是处理执行取舍：

- 调仓更频繁：响应快，但成本高。
- 调仓更慢：成本低，但信号滞后。
- 止损更紧：回撤小，但被噪声洗出。
- 止损更宽：持有趋势，但亏损更深。
- 仓位更高：收益高，但回撤和流动性压力高。
- 仓位更低：稳定，但资金利用率低。
