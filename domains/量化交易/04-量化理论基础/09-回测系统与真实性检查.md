# 回测系统与真实性检查

回测是用历史数据模拟策略在过去会怎么运行。它不是证明策略会赚钱，而是用历史对假设做压力测试。Zipline 是事件驱动回测系统，Backtrader 支持 broker simulation、订单、佣金、滑点和交易日历，bt 强调用可复用的策略逻辑块快速测试。这些框架共同说明：好回测必须模拟“当时能看到什么、能下什么单、能否成交”。

## 回测类型

| 类型 | 特点 | 适用 |
| --- | --- | --- |
| 向量化回测 | 用数组快速计算信号和收益 | 因子初筛、低复杂度策略。 |
| 事件驱动回测 | 按时间事件逐步推进，模拟订单和账户 | 接近实盘、复杂执行。 |
| 组合回测 | 关注权重变化和组合净值 | 资产配置、轮动、多因子。 |
| 订单级回测 | 模拟订单簿、成交和市场冲击 | 高频、执行算法。 |
| 走样本外回测 | 按时间滚动训练/验证 | 模型和参数选择。 |

## 回测引擎应模拟的对象

- 日历：交易日、节假日、半日市、夜盘。
- 数据：历史行情、复权、停牌、财报可见时间。
- 账户：现金、持仓、市值、保证金、杠杆。
- 订单：下单、撤单、部分成交、失败。
- 成本：佣金、税费、滑点、冲击。
- 风控：仓位限制、涨跌停、保证金不足。
- 记录：信号、目标仓位、真实成交、净值。

## 真实性检查清单

| 检查 | 问题 |
| --- | --- |
| 数据可见性 | 是否用了未来数据？ |
| 成交时点 | 信号生成时间是否早于下单时间？ |
| 成本模型 | 是否包含手续费、滑点、税费？ |
| 流动性 | 下单金额是否超过合理成交容量？ |
| 交易限制 | 停牌、涨跌停、T+1、最小交易单位是否处理？ |
| 复权 | 价格和收益计算口径是否一致？ |
| 现金 | 是否出现超买、负现金、保证金不足？ |
| 订单失败 | 失败订单是否记录并纳入复盘？ |
| 日志 | 是否能解释每次主要调仓？ |

## 回测不是实盘

回测天然乐观，因为：

- 历史数据清洗后比实盘数据干净。
- 回测通常不会模拟心理压力和暂停策略。
- 极端行情下流动性会消失。
- 参数是在看过历史后选择的。
- 实盘会遇到接口、网络、权限和风控限制。

因此，回测通过只是进入下一阶段，不是策略成功。

## 聚宽策略回测提示

Peter 使用聚宽时，应额外记录：

- `initialize` 中的基准、真实价格、avoid_future_data。
- 调度函数的运行时间。
- 每次调仓的候选、目标、订单和结果。
- 收益概述、交易详情、持仓收益和日志四件套。
- 本地静态检查和聚宽实际报错。
