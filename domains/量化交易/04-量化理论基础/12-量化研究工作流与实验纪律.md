# 量化研究工作流与实验纪律

量化研究需要像工程实验一样管理。Qlib 的工作流把数据、模型、回测和评估串起来；Machine Learning for Trading 强调从研究想法到可运行策略的迭代过程。对 Peter 的知识库来说，最重要的是把每次策略尝试变成可复盘的证据，而不是一串散落的回测结果。

## 标准工作流

1. 提出假设：一句话说明为什么应该赚钱。
2. 定义资产与宇宙：历史时点可复现。
3. 定义数据与特征：点时正确，记录缺失和异常处理。
4. 做信号验证：IC、分组收益、换手、分层。
5. 做简单策略回测：最少买卖器，验证方向。
6. 加入组合和执行约束：权重、成本、滑点、容量。
7. 做稳健性测试：年份、市场环境、成本、参数扰动。
8. 做样本外/滚动验证：防止只拟合开发期。
9. 生成报告：收益、风险、归因、失败点。
10. 决策：放弃、继续研究、模拟盘、实盘。

## 实验记录模板

| 字段 | 内容 |
| --- | --- |
| hypothesis | 策略假设。 |
| universe_rule | 交易宇宙生成规则。 |
| data_version | 数据来源和时间口径。 |
| signal_rule | 信号公式和窗口。 |
| portfolio_rule | 持仓数量、权重、约束。 |
| execution_rule | 调仓频率、订单、成本、滑点。 |
| risk_rule | 仓位、止损、回撤、杠杆限制。 |
| train_period | 开发/训练区间。 |
| test_period | 样本外验证区间。 |
| parameter_search | 搜索空间和失败版本。 |
| result | 核心指标和归因。 |
| decision | 下一步动作。 |

## 版本管理

策略研究中应保留：

- 原始策略代码。
- 每次修改后的代码。
- 回测导出文件。
- 日志和审计摘要。
- 分析报告。
- 参数变更记录。
- 放弃理由。

不要覆盖旧策略文件。Peter 之前要求“用另外一个 py 文件，不要覆盖之前的文件”是正确原则。

## Agent 行为规则

当用户要求“优化策略”时，Agent 不应直接调参数。先回答：

1. 当前策略假设是什么？
2. 当前失败来自哪一层？
3. 是否有数据偏差或过拟合风险？
4. 下一轮只改哪个机制？
5. 如何判断改动成功或失败？

只有这五个问题清楚，才写代码或跑回测。

## 从知识库到策略实验

知识库应服务于实验设计：

- 想研究选池，读市场/数据/因子/过拟合。
- 想研究买卖器，读组合/执行/回测。
- 想分析结果，读收益风险/绩效归因。
- 想防止过拟合，读样本外验证。
- 想写聚宽代码，读聚宽平台规范。
